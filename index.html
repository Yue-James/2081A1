MedTrack 总览 — 这个 App 是做什么的

Week 2Week 3Week 4

MedTrack 是一个帮助病人自我管理药物和症状的 Android App，用 Kotlin + Jetpack Compose 开发。

主要功能

  • 登录 / 注册（支持 CSV 预设用户 + 新注册用户）
  • 首页显示今日药物清单，可打勾标记已服用
  • 添加新药物，带完整表单验证
  • 记录症状，查看历史症状记录
  • Session 持久化：关掉 App 重开不需要重新登录
WelcomeScreen LoginScreen SignUpScreen HomeScreen AddMedicationScreen SymptomsScreen

App 使用流程

用户的完整操作路径

WelcomeScreen → LoginScreen / SignUpScreen → HomeScreen → AddMedicationScreen / SymptomsScreen

导航用 MainActivity 里的 currentScreen 字符串变量控制，when 块决定渲染哪个 Screen。每个 Screen 接收 lambda 回调来触发页面切换，而不是依赖 NavController。

MainActivity.kt — 整个 App 的导航中枢

Week 3 remember · mutableStateOf · Lambda 导航

MainActivity 是整个 App 唯一的 Activity，里面通过 setContent 承载完整的 Compose UI 树。它做三件事：

  • HD4 — Session 检查：读 SharedPreferences 的 logged_in_patient_id，决定初始页面是 "welcome" 还是 "home"
  • 导航状态：var currentScreen by remember { mutableStateOf(...) } 保存当前页面名称
  • 页面路由：when(currentScreen) 映射字符串到对应 Composable，每个 Composable 接收 lambda 来改变 currentScreen
val loggedInUser = sharedPref.getString("logged_in_patient_id", "")

var currentScreen by remember {
    mutableStateOf(if (loggedInUser.isNullOrEmpty()) "welcome" else "home")
}

when (currentScreen) {
    "welcome" -> WelcomeScreen(
        onLoginClick = { currentScreen = "login" },
        onSignUpClick = { currentScreen = "signup" }
    )
    "home" -> HomeScreen(
        onLogoutClick = {
            sharedPref.edit().remove("logged_in_patient_id").apply()
            currentScreen = "welcome"
        }
        // ...
    )
    // ...
}

登出逻辑：先从 SharedPreferences 删除 logged_in_patient_id，再把 currentScreen 设回 "welcome"，这样用户就必须重新登录才能进 Home（HD4 的 logout 部分）。

LoginScreen.kt — 登录验证 + CSV 解析

Week 3Week 4 remember · CSV 解析 · SharedPreferences

LoginScreen 实现了双来源登录验证：先查 CSV 里的预设用户，再查 SharedPreferences 里注册的新用户。同时，CSV 解析工具函数 readPatients 和 readMedications 也定义在这个文件里（可被其他 Screen 调用）。

CSV 解析原理（Week 4）

fun readPatients(context: Context): List<List<String>> {
    val patients = mutableListOf<List<String>>()
    context.assets.open("patients.csv").bufferedReader().useLines { lines ->
        lines.drop(1).forEach { line ->          // drop(1) 跳过标题行
            val row = line.split(",")            // 按逗号分割
            patients.add(row)
        }
    }
    return patients
}

返回值是 List<List<String>>，外层列表每个元素是一行，内层列表是该行的各个列。访问 row[0] 是 patientId，row[1] 是手机号，row[2] 是名字，row[3] 是密码。

登录逻辑三段 if-else

val csvPatient = csvPatients.find { it[1] == username }  // 查 CSV
val spUser = spUsers.find { it.phone == username }          // 查 SP

if (csvPatient == null && spUser == null) {
    errorMessage = "No account found with this phone number"
} else if (csvPatient != null) {
    if (csvPatient[3] != password) { errorMessage = "Incorrect password" }
    else { /* 保存 patientId 到 SP，跳转 Home */ }
} else if (spUser != null) {
    if (spUser.pass != password) { errorMessage = "Incorrect password" }
    else { /* 保存 patientId 到 SP，跳转 Home */ }
}

登录成功后，把当前用户的 patientId 写入 SharedPreferences（key: logged_in_patient_id），这就是 HD4 session 持久化的核心。

SignUpScreen.kt — 注册验证 + HD2 患者 ID 生成

Week 3Week 4 表单验证 · Gson · ID 生成算法

SignUpScreen 的核心职责：验证四个字段，生成唯一 patientId，把新用户存入 SharedPreferences。

表单验证规则

  • Full Name：不能为空
  • Phone：必须10位、以 "04" 开头、全是数字，且不能在 CSV 或 SP 里重复
  • Password：至少8位，包含至少一个字母和一个数字
  • Confirm Password：必须与 Password 完全一致

HD2 — 患者 ID 生成算法

val csvIds = readPatients(context).map { it[0] }  // 所有 CSV 里的 ID
val spIds = userList.map { it.patientId }            // 所有 SP 里的 ID
val allIds = csvIds + spIds

var maxIdNum = 1000
for (id in allIds) {
    if (id.startsWith("P")) {
        val num = id.substring(1).toIntOrNull()
        if (num != null && num > maxIdNum) { maxIdNum = num }
    }
}
val newId = "P" + (maxIdNum + 1).toString()   // e.g. "P1005"

这个算法扫描 CSV + SP 里所有已存在的 patientId，找到数字部分的最大值，+1 生成新 ID。保证新 ID 永远不与现有 ID 冲突。

HomeScreen.kt — 首页 & HD1 双来源合并

Week 3Week 4 Gson · filter · map · mutableStateListOf

HomeScreen 是登录后的核心页面，显示今日药物清单 + 导航按钮。最关键的实现是 HD1：把 CSV 药物和 SP 药物合并成一个统一列表。

患者名字查找（双来源）

先在 CSV 里按 patientId 查名字；如果找不到（是新注册用户），再去 SP 的 users 列表里找。

HD1 — 双来源药物合并

  • CSV 药物类型是 List<List<String>>
  • SP 药物类型是 List<MedicationItem>（data class）
  • 为了能在同一个 UI 循环里渲染，需要把 SP 的药物 map 成相同格式（List<String>），然后用 + 合并
val spMedications = spMedicationsRaw
    .filter { it.patientId == patientId }
    .map { med ->
        listOf(med.patientId, med.name, med.dosage,
               med.frequency, med.time, med.type, med.notes)
    }

val patientMedications = csvMedications + spMedications

Taken 勾选状态（内存状态）

val takenStates = remember(patientMedications.size) {
    mutableStateListOf<Boolean>().apply {
        repeat(patientMedications.size) { add(false) }
    }
}

mutableStateListOf 是 Compose 可响应的列表，修改某个 index 的值会触发 recomposition，让勾选即时生效（strikethrough 文字、计数更新）。注意这只是内存状态，切换 Screen 后会重置——spec 明确允许这样。

AddMedicationScreen.kt — 表单、Regex 验证 & SP 持久化

AddMedicationScreen 是一个可滚动的表单，包含七个字段。保存时先验证，验证通过再写入 SharedPreferences。

Dosage Regex 验证（重点，Band B）

val dosageRegex = Regex("^\\d+(\\.\\d+)?\\s*(mg|ml|g)$", RegexOption.IGNORE_CASE)

// 验证逻辑：
if (dosage.isBlank()) {
    dosageError = "Dosage cannot be empty"
    isValid = false
} else if (!dosage.matches(dosageRegex)) {
    dosageError = "Enter a valid dosage like 500mg or 10ml"
    isValid = false
}

正则表达式解析：^\d+（一个或多个数字）(\.\d+)?（可选的小数部分）\s*（可选空格）(mg|ml|g)$（单位，不区分大小写）。匹配 "500mg"、"10.5ml"、"2 G" 等。

Time Picker（TimePicker Dialog）

time 字段是 readOnly 的 OutlinedTextField，只能通过 TimePickerDialog 设置。用户点按钮，Dialog 弹出，选择后回调 time = "$hourText:$minuteText"，同时清除 timeError。

保存到 SharedPreferences 的流程

①读出现有 JSON → ②用 Gson 反序列化为 MutableList<MedicationItem> → ③加入新对象 → ④用 Gson 序列化回 JSON → ⑤写回 SP → ⑥Snackbar 提示 → ⑦延迟后 onBackClick() 返回 Home（Home 重新进入时会重读 SP，新药自动出现）。

var severity by remember { mutableFloatStateOf(5f) }
val severityInt = severity.toInt()

val severityColor = when (severityInt) {
    in 1..3 -> Color(0xFF4CAF50)  // 绿色 = Mild

Q1 · Reflection

Project Choices

"Can you explain the design and implementation choices you made in your project? What motivated these decisions?"

The biggest design choice I made was using a currentScreen string variable in MainActivity for navigation, rather than NavController. For six screens with simple linear flows, a when block that maps strings to composables is readable and easy to trace — you can always see all your routes in one place. Each screen receives lambda callbacks like onLoginSuccess = { currentScreen = "home" }, which keeps navigation logic centralised.

For data storage, I used SharedPreferences with Gson. The spec explicitly required SharedPreferences, and Gson is the natural companion for storing structured objects — it serialises a User or MedicationItem to a JSON string with one call, and deserialises it back just as easily.

I also completed all four HD extensions. HD1 (dual-source medication merge) was about converting two different data types into a common format so the Home screen could render them identically. HD4 (session persistence) checks SharedPreferences on launch to skip Login if the user is already authenticated.

Q2 · Reflection

Overcoming Challenges

"What were the major challenges you encountered during the project, and how did you address or overcome them?"

The biggest challenge was the HD1 dual-source medication merge. The problem was that CSV medications come back as List<List<String>> while SharedPreferences medications are List<MedicationItem> — a proper data class. To render both in the same UI loop, I had to normalise the format first.

I solved it by mapping each MedicationItem to a List<String> in the same column order as the CSV — listOf(med.patientId, med.name, med.dosage, ...). After that, csvMedications + spMedications produces a single unified list the UI can iterate with forEachIndexed.

The second challenge was the dosage regex. I needed to handle integers like "500mg", decimals like "10.5ml", optional whitespace between number and unit, and case-insensitive units. The regex ^\d+(\.\d+)?\s*(mg|ml|g)$ with IGNORE_CASE covers all these cases.

  • 解决方案：用 .map { med -> listOf(med.patientId, med.name, ...) } 把 MedicationItem 转换成 List<String>，列的顺序和 CSV 保持一致。

Q3 · Reflection

Lessons Learned

"Reflect on the project journey — what key lessons did you learn and how have they influenced your approach to problem-solving?"

The first lesson was how Compose recomposition works in practice. When you navigate back to the Home screen, the composable re-enters and re-reads SharedPreferences — so the newly added medication automatically appears without any extra refresh logic. I deliberately relied on this: since sharedPref.getString("medications", "[]") is called directly in the composable body, it always reflects the latest data on each recomposition.

The second lesson was to read specs carefully before writing code. For the Symptoms screen, the spec explicitly says the form "validates and shows a success message but does not persist the entry." I almost started implementing persistence — catching that line saved me unnecessary work.

The third lesson was about data format consistency. The HD1 merge taught me that when two sources have different shapes, you need to normalise one before combining — you can't just concatenate different types.

Error Handling

"How does your code handle error scenarios or edge cases? Can you provide an example of a specific strategy you used?"

Error handling works at the form level. Each field has a corresponding *Error state variable initialised to an empty string. When Save is clicked, all error variables are cleared first, then each validation rule runs. If a rule fails, its error string is set and isValid becomes false. Compose recomposes and shows the error text below the failing field. As soon as the user types again, onValueChange clears that field's error — so feedback is immediate both ways.

A specific example is the dosage field in Add Medication. The regex ^\d+(\.\d+)?\s*(mg|ml|g)$ handles the integer case, decimal case, optional whitespace, and case-insensitive units. If the field isn't blank but doesn't match, the error "Enter a valid dosage like 500mg or 10ml" appears.

In Sign Up, phone uniqueness is checked against both CSV and SharedPreferences — so a user who tries to re-register with the same phone gets "Phone number already registered" regardless of which source holds their account.

Design Trade-offs

"What trade-offs did you consider when making technical decisions, and how did these affect the functionality or performance of your code?"

The most significant trade-off was currentScreen vs NavController. The benefit is simplicity — all routes visible in one when block, callbacks are straightforward. The limitation is manual back-stack management. For HD4 logout, I handle this by explicitly clearing the SharedPreferences session and resetting to "welcome", so the state machine can't return to Home without re-authenticating.

The second trade-off was the in-memory taken toggles. mutableStateListOf holds the checkbox states, which reset when you navigate away. The spec explicitly says this is acceptable for A1 — persistence of taken state is deferred. The benefit is zero SharedPreferences writes per checkbox change, keeping the UI snappy.

For Symptoms, I chose not to persist new entries to SharedPreferences. Again, the spec says the form "does not persist the entry" in A1. The validation and Snackbar still give the user clear feedback that their input was accepted.

Complex Sections

"Identify a particularly complex section of your code — can you explain its functionality and why you chose this approach?"

The most complex section is the HD1 dual-source medication merge in HomeScreen. The challenge is that two data sources return different types: CSV medications are List<List<String>> and SharedPreferences medications are List<MedicationItem>.

The solution is to normalise the SharedPreferences type first. I use .filter to keep only the current patient's medications, then .map to convert each MedicationItem into a List<String> with the same column order as the CSV. After that, csvMedications + spMedications is a simple concatenation of two List<List<String>>. The UI loop uses forEachIndexed and accesses values with med.getOrNull(1) for name, med.getOrNull(2) for dosage, and so on.

The second complex section is the HD2 patient ID generation — scanning all existing IDs from both sources, finding the max numeric part, and incrementing it to guarantee uniqueness.

Welcome → Login / Sign Up → Home → Add Medication / Symptoms → Logout

2. 各 Screen 作用

Welcome 入口；Login 验证；SignUp 注册；Home 显示药物；AddMedication 保存新药；Symptoms 记录和显示症状。

3. 数据来源

CSV：patients.csv / medications.csv / symptoms.csv；SharedPreferences：users、medications、logged_in_patient_id。

4. Validation

Login：账号存在 / 密码正确；SignUp：姓名、手机号、密码、确认密码；AddMedication：name、dosage、frequency、time；Symptoms：date、symptom、severity、notes。

5. 关键实现思路

currentScreen + when 导航；SharedPreferences + Gson 持久化；CSV 只读；Home 合并 CSV 和 SP 药物；Symptoms 的 Slider 做 HD3 颜色反馈。

6. Week 2 / 3 / 4

Week 2：UI 组件和布局；Week 3：state、remember、screen switching；Week 4：SharedPreferences、Gson、CSV、dialogs、Regex、filter/map/reversed。

7. 最容易被问住

HD1 merge、dosage regex、为什么不用 NavController、为什么 Symptoms 不持久化、patient ID 生成逻辑。

8. 面试时最好主动提

“我刻意用了课程范围内最基础但合理的实现，因为这样最符合 assignment，也最容易清楚解释。”
