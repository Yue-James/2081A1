Q1

Project Choices

"Can you explain the design and implementation choices you made in your project? What motivated these decisions?"

I made several key design choices in MedTrack. First, I chose a single Activity with a currentScreen state variable for navigation instead of NavController, because at this stage of learning I was more comfortable with Compose state management from the lab materials, and it kept the navigation logic simple and easy to trace — each screen just receives lambda callbacks like onLoginSuccess = { currentScreen = "home" }.

For data storage, I used SharedPreferences with Gson to serialize objects as JSON arrays, because the spec explicitly required SharedPreferences, and Gson was the most natural way to store structured data like user accounts and medications without needing a database.

I also chose to implement all four HD extensions. For example, the dual-source medication merge was achieved by converting both CSV and SharedPreferences medications into the same List<String> format and concatenating them, so the Home screen renders everything with one unified list.

这题考什么： 你为什么这样设计，有没有主动思考过，而不是照抄 lab。

关键点拆解：

导航选择：你用的是 currentScreen state 变量，不是 NavController。原因是 Week1-4 的 lab 内容就是这样的，你熟悉这个模式。不要说"我不会 NavController"，要说"这个方案对当前项目的屏幕数量来说足够清晰"。

SharedPreferences + Gson：这是 spec 明确要求的，你要表现出是主动选择，而不是被动照做。说"spec 要求 SharedPreferences，Gson 是处理复杂对象最自然的方式"。

Q2

Overcoming Challenges

"What were the major challenges you encountered during the project, and how did you address or overcome them?"

The biggest challenge was implementing the dual-source medication merge for the HD extension. CSV medications are already in List<List<String>> format, but SharedPreferences medications are stored as List<MedicationItem> objects. I had to convert the SharedPreferences medications into the same format by mapping each MedicationItem to a listOf(patientId, name, dosage, frequency, time, type, notes), so both sources could be concatenated and rendered with the same card layout.

Another challenge was patient ID generation. I needed to scan both CSV and SharedPreferences for existing IDs, extract the numeric part after "P", find the maximum using toIntOrNull() for safe parsing, and increment it. I initialized the starting value at 1000 so that the first generated ID would be P1001, matching the CSV data pattern.

A smaller challenge was making the severity slider provide real-time visual feedback. I had to ensure the color and label update during dragging, not just on release, which meant deriving severityColor and severityLabel directly from the severity state so they recompose instantly on every onValueChange.

Q3

Lessons Learned

"Reflect on the project journey — what key lessons did you learn and how have they influenced your approach to problem-solving?"

One of the biggest lessons was understanding how Compose's state-driven UI works in practice. Early on I wasn't sure how to keep the medication list updated after adding a new one, but I realized that because the Home screen reads directly from SharedPreferences every time it recomposes, navigating back from Add Medication automatically triggers a recomposition and shows the new data. This taught me to think about when and why recomposition happens.

I also learned the importance of reading the spec carefully. For example, I initially thought I needed to persist the symptom entries immediately, but the spec explicitly says persistence for symptoms is deferred to Assignment 2. Reading the spec line by line saved me from over-engineering features that weren't required.

Another lesson was about data format consistency. When merging two data sources, if the formats don't match you get index out of bounds errors or display bugs. Standardizing everything to List<String> before combining it was a simple but important decision.

Adaptation and Improvement

"How did feedback or testing results impact your design decisions, and what changes did you implement as a result?"

During testing I noticed that if a user navigated to Add Medication and back without saving, the form still showed previously typed content. I added a Clear button specifically to handle this, and also made the Save button navigate back to Home after saving, which felt more natural than staying on the form.

I also tested edge cases in login — what if someone types a phone number that exists in the CSV but enters the wrong password, versus a number that doesn't exist at all. This led me to split the error messages into two distinct cases: "No account found with this phone number" and "Incorrect password", matching the spec's exact requirements.

When I added the HD extension for merging medications, I tested it by logging in as a CSV patient like P1001, adding a new medication, going back to Home, and verifying both the CSV medications and the newly added one appeared together.

Q5

Future Considerations

"If you could redo the project, what aspects would you change or improve upon, and why?"

If I could redo the project, I would organize the utility functions better. Currently readPatients and readMedications are defined inside LoginScreen.kt even though they're used by multiple screens. I'd move them to a separate Utils.kt file so the code is easier to navigate.

I'd also move data classes like User and MedicationItem to their own files rather than embedding them in screen files, since they're referenced across multiple screens.

For robustness, I'd add bounds checking when parsing CSV rows. Right now if a CSV row has fewer columns than expected, accessing an index like it[3] could cause an ArrayIndexOutOfBoundsException. Adding a check like if (row.size >= 4) before accessing columns would make it more defensive.

Finally, I would use NavController for navigation instead of a state variable, since NavController handles the back stack more explicitly and would make the HD4 session guard cleaner to implement.

Code Architecture

"Can you walk me through the overall structure of your code? Which parts do you consider most critical to the project's success?"

The app has a single Activity — MainActivity — which sets up the Compose content tree. Inside it, a currentScreen state variable acts as the navigation controller. A when block maps each screen name to its composable, and each composable receives lambda callbacks for navigation. For example, HomeScreen receives onAddMedicationClick = { currentScreen = "addMedication" }.

All six screens are in the screens package: WelcomeScreen, LoginScreen, SignUpScreen, HomeScreen, AddMedicationScreen, and SymptomsScreen. Data classes User and MedicationItem are defined in SignUpScreen.kt and AddMedicationScreen.kt respectively. Utility functions for CSV reading — readPatients, readMedications, readSymptoms — are in LoginScreen.kt and SymptomsScreen.kt.

The most critical parts are the login authentication logic in LoginScreen, the SharedPreferences read/write in AddMedicationScreen and SignUpScreen, and the dual-source merge in HomeScreen.

Q7

Error Handling

"How does your code handle error scenarios or edge cases? Can you provide an example of a specific strategy you used?"

I handle errors at multiple levels. For user input, I use field-level validation with inline error messages displayed directly below each input field. For example, in Sign Up, the phone number check has four separate conditions — must be 10 digits, must start with "04", must be all digits, and must be unique — and they all produce a single specific error string stored in phoneError state.

For null safety when reading from SharedPreferences, I use the Elvis operator throughout. For example: gson.fromJson(existingJson, type) ?: mutableListOf() ensures that if SharedPreferences has no data yet, I get an empty list instead of a NullPointerException.

For login, I explicitly handle three distinct cases: empty fields, phone not found in any data source, and phone found but wrong password. Each produces a different error message so the user knows exactly what went wrong.

For the dosage field, I use a regex ^\d+(\.\d+)?(mg|ml|g)$ to validate the format, and show "Must be number + mg/ml/g (e.g., 500mg)" if it fails.

Q8

Design Trade-offs

"What trade-offs did you consider when making technical decisions, and how did these affect the functionality or performance of your code?"

The main trade-off was choosing a state variable for navigation instead of NavController. The benefit is simplicity — the navigation logic is all in one when block that's easy to read and explain. The downside is that back stack management is implicit rather than explicit. In practice, since this is a single Activity with no real back stack, the system Back button exits the app rather than navigating between screens. For this assignment's scope, that behaviour is acceptable.

Another trade-off was reading SharedPreferences and CSV files directly inside composables rather than in a ViewModel. This means the data is re-read on every recomposition, which is slightly less efficient than caching it in a ViewModel. However, for the data sizes in this assignment — a few dozen rows at most — the performance impact is negligible, and it kept the code simpler and within the scope of what we'd covered in labs.

A third trade-off was not persisting the symptom entries in Band A. I chose to follow the spec exactly — which says persistence is deferred to Assignment 2 — rather than over-engineering a feature that wasn't required and might introduce bugs.

Q9

Complex Sections

"Identify a particularly complex section of your code — can you explain its functionality and why you chose this approach?"

The most complex section is the patient ID generation in SignUpScreen.kt. After the user passes all validation, the code collects all existing patient IDs from both the CSV file and SharedPreferences, then finds the maximum numeric ID, and generates the next one by incrementing.

Specifically, it takes all IDs as strings, filters for ones starting with "P", extracts the substring after "P", tries to parse it as an integer using toIntOrNull() — which returns null instead of throwing an exception if the format is unexpected — and tracks the maximum. The initial value is set to 1000 so that even if no valid IDs are found, the first generated ID is P1001, consistent with the CSV data pattern. This approach handles gaps in the ID sequence correctly — for example, if P1001, P1003, and P1005 exist, the next ID will be P1006, not P1004.

I chose this approach because it's robust to any combination of CSV and SharedPreferences users, and the logic is straightforward to trace step by step.

这题考什么： 你能不能解释最复杂的代码，逻辑清不清晰。

HD2 代码逐行解释（打开 SignUpScreen.kt 找这段）：

}

}

Refactoring Opportunities

"Given additional time or resources, what parts of your code would you refactor or optimize, and what benefits would these changes bring?"

The first thing I'd refactor is the placement of utility functions. readPatients and readMedications are currently in LoginScreen.kt but used by HomeScreen and SymptomsScreen as well. Moving them to a Utils.kt file would make the project easier to navigate and the dependencies clearer.

Second, I'd add defensive bounds checking to the CSV parsing. Currently the code does row[0], row[1], etc. without checking if the row has enough columns. If a CSV row is malformed — for example a trailing newline creates an empty row — this would cause an ArrayIndexOutOfBoundsException. Adding if (row.size >= expectedColumns) before accessing indices would prevent this.

Third, I'd extract SharedPreferences access into a single helper function or object rather than repeating context.getSharedPreferences("MedTrackPrefs", Context.MODE_PRIVATE) in every screen. This reduces the risk of typos in the key name causing hard-to-debug issues.

Q11

Project Summary

"Can you provide a concise summary of your project, highlighting the main features, challenges, and successes?"

MedTrack is a medication and symptom self-management app for patients. It has six screens: Welcome, Login, Sign Up, Home, Add Medication, and Symptoms.

The core flow is: patients log in using a phone number and password authenticated against a bundled CSV file or SharedPreferences for app-registered users. The Home screen displays today's medications filtered by patient ID, with checkboxes to mark each as taken and a live counter showing how many have been taken. New medications can be added and immediately appear on the Home screen alongside CSV medications. The Symptoms screen shows the patient's symptom history from CSV and allows logging new symptoms with a severity slider and date/time pickers.

I completed all of Band A and Band B, and all four HD extensions. The main challenge was the dual-source medication merge and patient ID generation. The biggest success is that the app runs end to end — login, add medication, see it on Home, log symptoms — without crashing.

"Describe a situation during the project where you had to adapt quickly — how did you handle it and what was the outcome?"

When I was implementing the Home screen to display medications, I initially planned to only show medications from SharedPreferences — the ones the user added in the app. But then I re-read the spec and realised that Band A requires loading medications from the CSV file filtered by patient ID. I adapted by converting the MedicationItem objects into List<String> to match the CSV format before concatenating, so the UI could render them identically.

The outcome was a clean unified list on the Home screen that seamlessly combines both sources, which is exactly what HD1 requires.
